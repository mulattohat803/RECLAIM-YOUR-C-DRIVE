01 //

THE PROBLEM

Running Kali Linux (or any distro) under WSL2? You've probably noticed your C: drive creeping toward 100% even though you haven't installed that much. Welcome to virtual disk bloat.

WSL2 stores your entire Linux filesystem inside a single file called ext4.vhdx. This container grows automatically as you use space — but it never shrinks back on its own, even after you delete files.

⚠
A Kali install using 58 GB of real data can have a .vhdx file consuming 129 GB on your Windows drive — wasting over 70 GB silently in the background.

Why Does This Happen?

When Linux deletes a file, it marks that space as free inside the Linux filesystem — but the .vhdx container on Windows has no idea. The container only knows the maximum size it ever reached, not what's currently occupied. This is called virtual disk bloat.

DIAGNOSE YOUR SYSTEM

Before doing anything, confirm you actually have this issue. Run the following inside your WSL terminal to see real disk usage:

kali@wsl2 ~ bash 
# Check actual Linux disk usage
$ df -h /

Then check the real size of the .vhdx file from PowerShell on Windows:

PowerShell — Administrator 
# Find and measure your Kali VHDX file
$pkg = Get-ChildItem "C:\Users\$env:USERNAME\AppData\Local\Packages" |
        Where-Object { $_.Name -like "*Kali*" }
$vhdx = Join-Path $pkg.FullName "LocalState\ext4.vhdx"
[math]::Round((Get-Item $vhdx).Length / 1GB, 2)
ℹ
If the .vhdx size is significantly larger than what df -h reports, you have virtual disk bloat. Continue with this guide to reclaim that space.

03 //

HOW THE FIX WORKS

The compaction process is a two-stage operation. Both stages are required — skipping Stage 1 means Stage 2 won't reclaim much space.

01

Zero-Fill Free Space (Inside Linux)

The dd command writes zeros across all free space inside your Linux filesystem. This marks empty blocks so the Windows compactor knows what to throw away. Without this, diskpart can't distinguish "empty" from "deleted data".

02

Compact the VHDX (From Windows)

Once free space is zeroed, Windows diskpart reads the zero regions and physically shrinks the .vhdx container to match actual usage. Think of it as vacuum-sealing a half-empty bag — contents stay the same, container gets smaller.

Stage 1 Command — Run in Kali

kali@wsl2 ~ bash 
# Zero-fill all free space (this is intentional — fills then deletes)
$ dd if=/dev/zero of=/tmp/zero.fill bs=1M 2>/dev/null; rm /tmp/zero.fill
⚠
Your disk will temporarily hit 100% capacity while dd runs. This is expected and intentional. The space returns as soon as the rm command clears the fill file. The screen may appear to freeze — leave it running.

How to Run

ℹ
Prerequisites: Complete the dd zero-fill step in Kali first. Plug in your laptop. Open PowerShell as Administrator.
PowerShell — Administrator 
PS> powershell -ExecutionPolicy Bypass -File "C:\Users\YourName\Downloads\compact-kali.ps1"

Full Script Source — compact-kali.ps1

compact-kali.ps1 
# ─────────────────────────────────────────────────────────
# compact-kali.ps1  //  WSL2 VHDX Compaction Script
# Run as Administrator after dd zero-fill inside Kali
# ─────────────────────────────────────────────────────────

$vhdxPath = (Resolve-Path "C:\Users\$env:USERNAME\AppData\Local\Packages\KaliLinux*\LocalState\ext4.vhdx").Path
$targetGB = 60   # Adjust to your expected final size

# Step 1: Shut down WSL
Write-Host "`n[1/4] Shutting down WSL..." -ForegroundColor Cyan
wsl --shutdown
Start-Sleep -Seconds 3

# Step 2: Get starting size
$startSize = (Get-Item $vhdxPath).Length / 1GB
Write-Host "[2/4] Compacting $([math]::Round($startSize,2)) GB VHDX..." -ForegroundColor Cyan

# Step 3: Build and run diskpart script silently
$diskpartScript = @"
select vdisk file="$vhdxPath"
attach vdisk readonly
compact vdisk
detach vdisk
exit
"@
$tempScript = "$env:TEMP\diskpart_compact.txt"
$diskpartScript | Out-File -FilePath $tempScript -Encoding ASCII
$job = Start-Process diskpart -ArgumentList "/s `"$tempScript`"" -PassThru -WindowStyle Hidden

# Step 4: Live progress monitor
while (-not $job.HasExited) {
    $current  = (Get-Item $vhdxPath).Length / 1GB
    $reduced  = $startSize - $current
    $pct      = [math]::Min(100, [math]::Round([math]::Max(0,($reduced/($startSize-$targetGB)))*100,1))
    $bar      = "#" * [math]::Round($pct/2) + "-" * (50-[math]::Round($pct/2))
    Clear-Host
    Write-Host "VHDX Compaction Progress" -ForegroundColor Cyan
    Write-Host "Current: $([math]::Round($current,2)) GB  |  Saved: $([math]::Round($reduced,2)) GB"
    Write-Host "[$bar] $pct%" -ForegroundColor Green
    Write-Host "Updated: $(Get-Date -Format 'HH:mm:ss')"
    Start-Sleep 5
}

# Done!
$final = (Get-Item $vhdxPath).Length / 1GB
Write-Host "Done! $([math]::Round($startSize,2)) GB -> $([math]::Round($final,2)) GB  |  Saved: $([math]::Round($startSize-$final,2)) GB" -ForegroundColor Green
Remove-Item $tempScript -ErrorAction SilentlyContinue

PREVENT FUTURE BLOAT

Run this maintenance routine every few months, or after any major cleanup inside Kali:

kali@wsl2 ~ bash — Periodic Maintenance 
# 1. Clean apt cache and orphaned packages
$ sudo apt autoremove --purge -y && sudo apt clean

# 2. Zero-fill free space for next compaction
$ dd if=/dev/zero of=/tmp/zero.fill bs=1M 2>/dev/null; rm /tmp/zero.fill

# 3. Then run compact-kali.ps1 from Windows

Optional: WSL Config Tuning

Add this to C:\Users\YourName\.wslconfig for better memory management:

C:\Users\YourName\.wslconfig 
# WSL2 memory and swap settings
[wsl2]
autoMemoryReclaim=gradual
memory=8GB          # limit RAM usage (adjust to your system)
swap=2GB
ℹ
Note: autoMemoryReclaim helps with RAM but does not auto-compact the disk. Manual compaction using compact-kali.ps1 remains the most reliable method for reclaiming disk space.
